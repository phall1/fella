# fella Threat Model

> **Last updated:** 2026-06-07  
> **Version:** 0.4.0 "Chain"  
> **Status:** Living document — evolves with every release.

This document describes the adversaries fella is designed to resist, the attacks it mitigates, and the gaps it does **not** currently close. If you operate under a threat model stronger than what is documented here, assume fella will fail.

---

## 1. Asset Inventory

| Asset | Location | Protection Goal |
|-------|----------|-----------------|
| Session state (identities, backend config, keys) | `/var/lib/fella` | Confidentiality + availability during session |
| Identity backups (hostname, machine-id, timezone, locale) | `/var/lib/fella/original` | Integrity (must restore cleanly) |
| Tor data directory | `/var/lib/fella/tor` | Confidentiality, forward secrecy on session end |
| Active network traffic | `fella` netns via `torsocks` | Unlinkability from real identity / origin IP |
| Process identity | `/proc/<pid>` | Obfuscation from casual inspection |
| Binary itself | `/usr/local/bin/fella` | Integrity (best-effort) |

## 2. Adversaries

### 2.1 Local Network / ISP (Passive Observer)
**Capabilities:** Sees all cleartext metadata leaving the machine: packet sizes, timing, destination IPs, DNS queries, SNI.

**Assumption:** The adversary cannot break Tor cryptography or control a significant portion of Tor relays.

**Mitigations:**
- All application traffic forced through Tor SOCKS5 inside isolated netns.
- DNS queries sent through Tor DNSPort.
- Fail-closed iptables killswitch (strict mode drops everything except Tor).
- Constant-rate padding subagent emits fixed-size packets at fixed intervals through the tunnel.
- MAC address rotation breaks L2 / DHCP correlation per session.

**Gaps:**
- Cover traffic is generated by `curl` POSTs — not perfectly constant-rate at the wire level. An attacker with sub-millisecond timing resolution can still distinguish true application bursts from padding.
- No fake ACK / fake TLS handshake padding.

### 2.2 Nation-State Firewall (Active Blocker)
**Capabilities:** Blocks known Tor relays and directory authorities. Performs DPI to fingerprint Tor / obfs4 / snowflake.

**Mitigations:**
- Auto-detects `obfs4proxy` or `snowflake-client` and injects bridge lines into `torrc`.
- Embeds default obfs4 bridge lines for censored regions.
- Supports custom bridge configs at `/var/lib/fella/bridges.conf`.

**Gaps:**
- If obfs4/snowflake are not installed, fella falls back to direct Tor and will likely fail in censored countries.
- No active probing resistance beyond what Tor/bridges already provide.
- No domain-fronting fallback beyond snowflake defaults.

### 2.3 Endpoint Forensics (Post-Session)
**Capabilities:** Gains physical or remote access to disk after fella has stopped. Looks for artifacts in `/var/lib/fella`, logs, shell history, packet captures.

**Mitigations:**
- `fella wipe` performs 3-pass overwrite (random → complement → random) + `fsync`.
- `fella start --ephemeral` mounts `/var/lib/fella` as a tmpfs; all session state lives in RAM and evaporates on reboot / power loss.
- Identity restore on `stop` reverts hostname, machine-id, timezone, locale.
- Secure memory: `mlock` + `MADV_DONTDUMP` + explicit `secureZero`.
- Encrypted state file via `init --encrypt` + `FELLA_PASSPHRASE`.

**Gaps:**
- RAM freeze / cold-boot attacks can dump tmpfs contents before power loss.
- Swap may contain fella pages unless swap is encrypted or disabled.
- Kernel logs (`dmesg`, auditd, systemd journal) may record MAC changes, namespace creation, iptables modifications.
- The fella binary in `/usr/local/bin/fella` is itself evidence of installation.

### 2.4 Local Process Compromise
**Capabilities:** Malware running as the same UID or root can inspect `/proc`, ptrace, inject, or exfiltrate.

**Mitigations:**
- seccomp-bpf deny-list blocks `ptrace`, `process_vm_*`, `userfaultfd`, `kexec_*`, module loading, `keyctl`, `bpf`, `open_by_handle_at`, `perf_event_open`.
- `PR_SET_NO_NEW_PRIVS` prevents setuid privilege escalation in children.
- Process name masquerade (`prctl(PR_SET_NAME)`) makes fella look like a common systemd service in `ps` / `top`.

**Gaps:**
- `prctl(PR_SET_NAME)` is trivially defeated by reading `/proc/<pid>/exe` or `/proc/<pid>/cmdline`.
- seccomp does not block all exfiltration channels (e.g., legitimate network writes, signals, shared memory with non-fella processes).
- If the attacker already has root, seccomp can be disabled.

### 2.5 Container / Hypervisor Fingerprinting
**Capabilities:** Remote service detects that the client is running inside a container or VM via `/proc/cpuinfo`, `/proc/version`, kernel params, uptime, etc.

**Mitigations:**
- Bind-mount fake `/proc/cpuinfo`, `/proc/version`, `/proc/uptime`, `/proc/stat`.
- LD_PRELOAD `libfella.so` intercepts `uname()` and returns spoofed values.
- Writable `/proc/sys/kernel/*` parameters are rewritten.

**Gaps:**
- Some kernel params are read-only in LXC; fella silently skips them.
- Browser fingerprinting (canvas, WebGL, fonts, timezone via JS) is not yet mitigated.
- No VM-level clock skew normalization.

### 2.6 Global Passive Adversary (NSA / Five Eyes)
**Capabilities:** Controls or observes large fractions of Tor relays, backbone fibers, CDN edges. Correlates traffic across multiple vantage points using timing, volume, and entry/exit heuristics.

**Mitigations:**
- Tor's existing three-hop circuit design.
- Constant-rate padding increases the noise floor.
- VPN → Tor chaining adds an extra hop before Tor entry.
- MAC rotation prevents link-layer correlation across sessions.

**Gaps:**
- **This is the adversary fella is least equipped to resist.** A global passive adversary with sufficient coverage can perform end-to-end confirmation attacks that no client-side tool can fully defeat.
- fella does not implement adaptive padding, waterfall padding, or loopcover-style traffic shaping.
- No guard rotation strategy beyond Tor defaults.
- No support for client authorization / onion services.

## 3. Trust Assumptions

1. The Linux kernel is trustworthy and not backdoored.
2. The underlying hardware is trustworthy (no Intel ME / AMD PSP / BMC compromise).
3. `tor`, `torsocks`, `iptables`, `iproute2`, `wg`, and `obfs4proxy` are not backdoored.
4. The operator does not install unrelated malware during the fella session.
5. The operator's destination (e.g., the website they visit) does not collude with the adversary.

## 4. Attack Matrix

| Attack | Status | Notes |
|--------|--------|-------|
| ISP sees origin IP of HTTP request | **Mitigated** | All traffic routed through Tor or WireGuard → Tor. |
| ISP sees DNS query for destination | **Mitigated** | DNSPort sends queries over Tor. |
| Firewall blocks TCP/9050 | **Mitigated** | obfs4 / snowflake bridges used if installed. |
| DPI fingerprints unencrypted Tor | **Mitigated** | Bridges hide Tor handshake. |
| Local `ps` reveals "fella" | **Mitigated** | Process masquerade renames comm. |
| `/proc/<pid>/exe` reveals fella | **Partial** | Process name faked; executable path still visible. |
| Post-session disk forensics | **Mitigated** | Wipe + ephemeral tmpfs + encrypted state. |
| Cold-boot attack on RAM | **Not mitigated** | Physical seizure while powered is catastrophic. |
| Compromised Tor exit node | **Partial** | Tor only; use HTTPS + certificate pinning. |
| Global passive correlation | **Not mitigated** | Fundamentally hard; padding helps but is not a cure. |
| Browser fingerprinting | **Not mitigated** | Planned for v0.5. |
| Side-channel (cache, power, EM) | **Not mitigated** | Out of scope. |
| Evil maid / boot tampering | **Not mitigated** | Out of scope. |

---

## 5. Operational Security Recommendations

- Use `--ephemeral` for any session where physical seizure is plausible.
- Combine with full-disk encryption and encrypted swap.
- Run in a disposable VM or live USB, not on a daily driver.
- Keep bridge configs (`/var/lib/fella/bridges.conf`) up to date.
- Do not reuse identities across sessions — `fella rotate` is cheap.
- Wipe the session (`fella wipe`) before shutdown, even with `--ephemeral`.
- Use the browser bundle or wait for v0.5 browser isolation; regular Firefox leaks fingerprints.
